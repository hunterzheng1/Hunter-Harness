# 技能中心设计文档

## 1. 目标

技能中心用于管理可被不同 AI Agent 使用的 Skill。新的设计以“标签 + Agent 适配 + 检查发布”为核心，不再把“分类”作为业务概念。上传后的 Skill 先进入未发布状态，用户可以进入详情页查看文件内容、使用示例、系统配置、检查与发布、版本记录；发布后才成为 `npx @hunter-harness/skill-cli install <skill> --agent <agent>` 可下载的正式版本。

本设计对齐当前前端改造方向：

- 技能列表不再展示或过滤“分类”。
- 标签是主要组织方式，支持多选过滤。
- 状态只区分“已发布 / 未发布”。
- 上传不再走复杂的 Skill 提案页面，而是进入本地暂存的未发布 Skill。
- 检查与发布独立为详情页 Tab，检查结果只提示风险，不阻拦发布。
- 同一个 Skill 可以有多个 Agent 适配版本，并可以设置默认 Agent。
- AI 配置作为左侧栏独立入口，后续用于选择供应商、模型、启用状态、连通性测试和使用统计。

## 2. 页面信息架构

### 2.1 左侧导航

新增入口：

- AI 配置：配置供应商、模型、启用状态、连通性测试、使用统计。

保留入口：

- 总览
- 项目
- 工作流
- 技能
- 审核队列
- 制品

### 2.2 技能列表页

主区域：

- 搜索框：按名称、slug、描述搜索。
- Agent 下拉：按已适配 Agent 过滤。
- 状态下拉：全部 / 已发布 / 未发布。
- 标签多选区：展示所有有效标签，点击选中，再次点击取消，可多选。
- 列表分页：按页面可读尺寸分页，避免一次展示过多 Skill。
- 技能行信息：
  - 名称
  - 描述
  - 标签
  - 版本
  - 适配 Agent 数
  - 被工作流使用次数
  - 更新时间
  - 发布状态
  - 删除按钮，点击后弹出确认。

右侧栏：

- 上传技能：
  - 支持 ZIP 或文件夹。
  - 上传后新增一个未发布 Skill。
  - 不再创建“待审核提案”。
- 技能统计：
  - 技能总数
  - 已发布数
  - 未发布数
  - 有效标签数
  - 已配置 Agent 数
  - 被工作流使用的 Skill 数

### 2.3 技能详情页

Tab 结构：

- 文件内容：展示当前 Skill 文件包；没有原始文件包时展示 `skill-ir.json` fallback。
- 使用示例：展示 2-3 个典型使用方式，包括请求方式、输入、预期输出、相关文件。
- 系统配置：展示基本信息、工具适配、允许能力、执行要点、契约与安全边界。
- 检查与发布：上传暂存包、执行检查、查看当前版本与暂存版本 diff、发布新版本。
- 版本记录：左侧历史版本，右侧当前版本与上一版本 diff；发布时记录变更信息。

系统配置中不再展示分类。基本信息只允许直接编辑：

- 描述
- 标签

以下信息保留为系统提取或适配结果：

- 名称
- 版本
- 工具适配 Agent
- 允许能力
- 执行要点
- 触发器
- 输入
- 输出
- 禁止动作
- 必要上下文
- 来源

## 3. 数据模型设计

### 3.1 Skill

建议服务端长期模型：

```ts
type SkillRecord = {
  skillId: string;
  slug: string;
  name: string;
  description: string;
  tags: string[];
  status: "published" | "unpublished";
  latestVersion: string | null;
  defaultAgent: AgentId | null;
  agents: AgentSkillConfig[];
  createdAt: string;
  updatedAt: string;
};
```

说明：

- 移除 `category`。
- `status` 简化为已发布 / 未发布。
- `latestVersion` 表示默认 Agent 或全局主版本的最新发布版本。
- 不同 Agent 可有独立版本。

### 3.2 Agent 适配版本

```ts
type AgentSkillConfig = {
  agent: "claude-code" | "codex" | "cursor" | "generic";
  enabled: boolean;
  isDefault: boolean;
  installTarget: string;
  latestVersion: string | null;
  draftVersion: string | null;
  sourcePackagePath: string | null;
  checkProfile: AgentCheckProfile;
};
```

规则：

- 只有一个 Agent 时自动作为默认 Agent。
- 多个 Agent 时允许用户切换默认 Agent。
- 用户进入未配置 Agent 页面时，展示横幅提示当前配置来自默认 Agent，但仍允许上传该 Agent 专属包。
- `npx install --agent <agent>` 下载到对应工具目录。例如 Cursor 下载到 `.cursor` 相关路径。

### 3.3 标签

```ts
type SkillTag = {
  tagId: string;
  slug: string;
  label: string;
  active: boolean;
  usageCount: number;
};
```

标签是 Skill 的唯一分组方式。列表页显示所有 active 标签，支持多选过滤。

## 4. 上传与发布流程

### 4.1 上传

上传入口在列表页和详情页检查发布 Tab 中都可存在：

1. 用户上传 ZIP 或文件夹。
2. 系统解包并识别 `SKILL.md`、`references/`、`scripts/`、`assets/` 等文件。
3. 系统提取基本信息：
   - 名称
   - 描述
   - 版本
   - 标签候选
   - Agent 适配信息
   - 允许能力
   - 执行要点
   - 契约与安全边界
4. 新增或更新为未发布 Skill。
5. 用户进入详情页继续检查和发布。

### 4.2 检查

点击检查时不直接开始，而是弹出确认窗口，说明检查范围：

- 结构完整性
- Agent 规范
- 权限与安全
- 使用示例质量
- 系统配置完整性
- 跨 Agent 适配差异

确认后生成检查结果：

- 绿灯：通过
- 黄灯：建议修改
- 红灯：高风险或明显错误

红灯发布时提醒，但不阻拦发布。

### 4.3 发布

点击发布弹出发布信息窗口：

- 当前版本号
- 新版本号，默认补丁版本 +0.0.1，可手动调整。
- 增加文件数
- 修改文件数
- 变动行数
- 变更信息，可手动填写，也可 AI 生成。

确认发布后：

- 暂存草稿变为当前 Agent 的最新发布版本。
- 更新版本记录。
- `npx @hunter-harness/skill-cli install <skill> --agent <agent>` 下载到该 Agent 的最新发布版本。

## 5. 规则与校验方案

参考资料：

- `requirements/agent规则/Claude Code Skill 规范与要求研究报告.md`
- `requirements/agent规则/Codex 与 Claude Code 技能开发规范与要求研究报告.md`
- `requirements/agent规则/Claude Code、Codex、Cursor 三个平台开发 Skill 与 Plugin 与 Extension 的规范与要求对比研究.md`
- `requirements/agent规则/昆仑智码-skill-creator`

当前先配置 4 个 Agent：

- Claude Code
- Codex
- Cursor
- Generic

### 5.1 程序校验

适合程序校验的规则必须确定、可重复、可解释：

| 检查项 | 程序校验内容 | 失败等级 |
|---|---|---|
| 入口文件 | 是否存在 `SKILL.md` | 红 |
| 文件路径 | 是否包含 `..`、绝对路径、危险隐藏文件 | 红 |
| ZIP 安全 | 是否 zip slip、超大文件、异常文件名 | 红 |
| frontmatter | YAML 是否可解析 | 红 |
| 命名 | slug 是否 kebab-case | 黄/红 |
| 描述 | 是否为空、是否超长 | 黄/红 |
| 结构 | 是否识别 `references/`、`scripts/`、`assets/` | 绿/黄 |
| 权限 | 是否出现 `Bash(*)`、危险命令、未声明网络访问 | 黄/红 |
| 敏感信息 | 是否出现 token、secret、private key、内网固定凭证 | 黄/红 |
| Agent 目标路径 | 是否能映射到 `.claude/skills`、`.codex`、`.cursor` 等路径 | 红 |
| 版本 | semver 是否有效，新版本是否大于当前版本 | 黄/红 |

### 5.2 AI 分析

适合 AI 分析的规则需要语义判断：

| 检查项 | AI 分析内容 | 失败等级 |
|---|---|---|
| 触发描述质量 | description 是否清楚说明做什么、何时用 | 黄 |
| 正文质量 | 是否有清晰步骤、输入、输出、异常处理 | 黄 |
| 使用示例 | 示例是否覆盖 2-3 个真实场景 | 黄 |
| 系统配置提取 | 能否从正文提取 allowed capabilities、instructions、contracts | 黄 |
| 跨 Agent 适配 | 是否引用了目标 Agent 不支持的能力，例如 subagent、hook、MCP 等 | 黄/红 |
| 安全边界 | 是否把副作用动作写成自动触发 | 红 |
| 修复建议 | 给出具体文件、具体行、建议改法 | 黄/红 |
| 变更信息 | 根据 diff 生成发布说明 | 绿 |

### 5.3 组合校验

最终检查结果应由程序和 AI 组合：

1. 程序先做确定性解析和安全扫描。
2. AI 读取程序产物和 Skill 文件内容，做语义分析。
3. 系统把结果合并为统一检查项。
4. 一键修复只对可安全确定的内容执行，例如：
   - 修复 frontmatter 字段名。
   - 补充缺失标签候选。
   - 生成使用示例草稿。
   - 收窄明显危险的权限文案。
5. 涉及语义大改或删除内容时，只生成 patch 预览，不自动覆盖。

## 6. 四类 Agent 规则概要

### 6.1 Claude Code

程序规则：

- 推荐 `SKILL.md` 入口。
- 支持 `.claude/skills/<skill>/SKILL.md`。
- 检查 `allowed-tools`、`disallowed-tools`、`disable-model-invocation`。
- 对高副作用 Skill 检查是否手动触发。

AI 规则：

- 判断是否过度依赖 hooks/subagents。
- 判断正文是否适合 Claude Code 的渐进加载。
- 判断权限描述是否过宽。

### 6.2 Codex

程序规则：

- 采用 Agent Skills 目录结构。
- 检查 `SKILL.md` 和 kebab-case 命名。
- 检查是否依赖 Codex 不支持或当前环境未启用的 hook/subagent 能力。
- 检查权限 profile、MCP 工具、网络访问描述是否清晰。

AI 规则：

- 判断 Claude Code 专属字段是否需要改写为 Codex 指令。
- 判断是否可在当前会话内执行，还是需要外部 MCP/App Server。
- 判断使用示例是否适合 Codex 的显式任务流程。

### 6.3 Cursor

程序规则：

- 支持 `.cursor/rules`、Cursor skills/plugin 结构的适配输出。
- 检查目标路径、规则文件名、frontmatter。
- 检查是否误用了 Claude Code 专属路径。

AI 规则：

- 判断 Skill 内容是否更适合 Cursor Rule、Command 还是 Skill。
- 判断是否需要改写为 IDE 上下文规则。
- 判断是否存在不适合 Cursor Agent 的自动副作用。

### 6.4 Generic

程序规则：

- 只要求标准 Markdown Skill 包。
- 不允许依赖特定 Agent 的专属字段作为唯一执行路径。
- 检查基本目录、路径安全、敏感信息。

AI 规则：

- 把工具专属能力改写为通用操作步骤。
- 标注需要人工替换的工具能力。
- 输出可迁移性建议。

## 7. 一键修复设计

一键修复不是“自动发布”。它只生成修复草稿，并在 diff 中展示：

- 当前版本文件
- 修复后文件
- 变动文件数
- 变动行数
- 风险变化

修复分级：

- 安全自动修复：格式、frontmatter、标签候选、明显 typo。
- 需要确认修复：权限收窄、自动触发改手动触发、示例重写。
- 只建议不修复：删除大段正文、改变核心业务逻辑、替换 API 地址。

## 8. 版本策略

同一个 Skill 支持全局元数据和 Agent 独立版本：

- 全局元数据：名称、描述、标签、默认 Agent。
- Agent 版本：每个 Agent 有自己的 latestVersion、draftVersion、文件包、检查结果。
- 默认 Agent：没有专属配置的 Agent 使用默认 Agent 的发布版本作为回退来源。
- 发布时只发布当前 Agent 的草稿，不影响其他 Agent。

版本记录展示：

- 左侧历史版本列表。
- 右侧展示选中版本与上一版本 diff。
- 展示发布时填写或 AI 生成的变更信息。

## 9. CLI 下载策略

命令形式：

```bash
npx @hunter-harness/skill-cli install <skill-slug> --agent <agent>
```

说明：

- `@hunter-harness/skill-cli` 是一个统一安装器。
- 不需要每个 Skill 单独发 npm 包。
- 所有 Skill 都通过 registry API 下载。
- CLI 根据 `--agent` 决定安装路径和适配产物。
- 若指定 Agent 没有专属版本，则使用默认 Agent 的版本并提示 fallback 来源。

## 10. 当前后端功能覆盖

本节对齐当前代码中的真实后端能力，作为实现核对清单。权威来源包括 `apps/server/src/app.ts`、`apps/server/src/registry/store.ts`、`apps/server/openapi/hunter-harness-v1.yaml`、`packages/contracts/src/registry.ts` 和 `packages/skill-cli/src/bin.ts`。

### 10.1 Skill Registry 读接口

| 能力 | API / 模块 | 描述 |
|---|---|---|
| 技能列表 | `GET /api/v1/skills` | 返回 canonical Skill registry；支持 search、tag、agent、status 查询，后端仍保留 category 查询兼容字段，前端不再把 category 作为主要业务筛选。 |
| 技能详情 | `GET /api/v1/skills/{slug}` | 返回 Skill 基本信息、标签、状态、最新版本、适配 Agent、revision 和 canonical Skill IR。 |
| 版本记录 | `GET /api/v1/skills/{slug}/versions` | 按时间倒序返回已发布版本、对应 IR、source proposal 和 artifacts。 |
| 适配预览 | `GET /api/v1/skills/{slug}/adapter-preview/{agent}` | 返回已验证发布版本的 adapter 输出预览、目标路径、source IR hash 和 compiler version；MVP 仅 Claude Code 可安装，其他 Agent 作为 contract-only 边界。 |
| Skill artifact 列表 | `GET /api/v1/skill-artifacts` | 返回所有已发布 Skill adapter artifacts，用于制品历史和审计展示。 |
| ZIP 下载 | `GET /api/v1/skills/{slug}/artifacts/{agent}/download` | 下载已批准 adapter ZIP，响应包含 `Content-Disposition`、`X-Content-SHA256` 和 `ETag`，并写入 `skill.artifact.downloaded` 审计。 |

### 10.2 Skill proposal、审核与发布

| 能力 | API / 模块 | 描述 |
|---|---|---|
| 提案列表 | `GET /api/v1/skill-proposals?status=...` | 返回 Skill proposal 队列，可按状态过滤。 |
| 提案详情 | `GET /api/v1/skill-proposals/{proposal_id}` | 返回 proposal、validation、审核时间和 proposed IR。 |
| 创建提案 | `POST /api/v1/skill-proposals` | 接收 `skill_ir` 与 `agent`，执行 schema 校验、敏感信息扫描、Claude Code 编译校验、版本递增校验，创建 `pending_review` proposal。MVP 只允许 `claude-code` 发布。 |
| 审核提案 | `POST /api/v1/skill-proposals/{proposal_id}/review` | owner 审核 approve/reject；approve 时发布新版本、生成 Claude Code ZIP artifact、更新 latest version 和 Skill detail；reject 不改变 canonical Skill。 |
| 幂等与审计 | `mutation(...)` + `writeAudit(...)` | 写接口走认证、request id、idempotency key 和审计事件；创建、批准、拒绝、下载均有明确 action。 |
| 校验模型 | `registryValidationSchema` | 当前 validation 包含 schema_valid、sensitive_findings、claude_compilable；后续程序检查器和 AI 分析器可扩展更细的检查项。 |

### 10.3 标签直接维护

| 能力 | API / 模块 | 描述 |
|---|---|---|
| 标签列表 | `GET /api/v1/tags` | 返回标签库，包含 active、revision、created_at、updated_at。 |
| 创建标签 | `POST /api/v1/tags` | 直接创建标签并写入 `tag.created` 审计，不创建 proposal。 |
| 更新标签 | `PATCH /api/v1/tags/{tag_id}` | 支持重命名或启停，使用 revision 做乐观并发控制，写入 `tag.updated` 审计。 |
| 合并标签 | `POST /api/v1/tags/{tag_id}/merge` | 将 source 标签绑定迁移到 active target 标签，source 自动停用，写入 `tag.merged` 审计。 |
| 绑定/解绑 Skill 标签 | `PUT/DELETE /api/v1/skills/{slug}/tags/{tag_id}` | 直接更新 Skill tags 和 revision，写入 `skill.tag.bound` 或 `skill.tag.removed` 审计。 |

### 10.4 Workflow 与项目绑定

| 能力 | API / 模块 | 描述 |
|---|---|---|
| Workflow 列表/详情 | `GET /api/v1/workflows`、`GET /api/v1/workflows/{workflow_id}` | Workflow 是有序 Skill 清单，不是 DAG runner；包含 key、profile、default_agent、enabled、skill_slugs 和 revision。 |
| 创建/更新 Workflow | `POST /api/v1/workflows`、`PATCH /api/v1/workflows/{workflow_id}` | 直接审计写入；校验 Skill 不重复、引用 Skill 必须已发布、支持 default_agent，并启用对应 profile。 |
| 删除 Workflow | `DELETE /api/v1/workflows/{workflow_id}?revision=...` | 带 revision 检查；若仍被项目绑定则返回 `WORKFLOW_IN_USE`，避免删除受引用 Workflow。 |
| 项目 Workflow 绑定 | `GET/PUT /api/v1/projects/{project_id}/workflow-binding` | 项目可直接绑定一个 Workflow，使用 revision 做并发控制，并写入 `project.workflow.bound` 审计；项目文件治理仍走原 proposal/review/update 协议。 |

### 10.5 数据合同、持久化与 CLI

| 能力 | 文件 / 模块 | 描述 |
|---|---|---|
| Registry 合同 | `packages/contracts/src/registry.ts` | 定义 RegistryAgent、Skill status、proposal status、artifact、version、tag、workflow、project workflow binding 等 schema。当前合同仍包含 category 字段作为兼容数据，产品层以 tags 为主。 |
| Registry Store | `apps/server/src/registry/store.ts` | 维护 skills、proposals、tags、workflows、projectBindings；发布时生成版本与 artifact；所有写入后可通过 persistence 保存。 |
| 持久化 | `apps/server/src/registry/persistence.ts` + migration | PostgreSQL 使用 canonical registry JSONB snapshot 保存 registry state，MVP 后续可迁移到规范化表。 |
| Skill CLI install | `packages/skill-cli/src/bin.ts` | `install` 通过 registry artifact download 获取 ZIP，校验 SHA-256 和 ZIP identity，原子写入 `.claude/skills/<slug>/`，默认拒绝覆盖本地 dirty skill。 |
| Skill CLI upload | `packages/skill-cli/src/bin.ts` | `upload <directory-or-zip> --agent claude-code` 从目录、ZIP 或单个 IR 文件中解析 canonical Skill IR，并创建 Skill proposal；当前校验目标仅 Claude Code。 |

### 10.6 当前实现边界

- Web 列表页的上传交互目前用于本地 demo 暂存；真实后端发布链路以 Skill proposal + owner review 为准。
- 服务端当前可发布 adapter 只覆盖 Claude Code；Codex、Generic、MCP 在 UI 和合同中保留为目标 Agent，但后端发布时仍返回 contract-only / unsupported。
- 后端合同仍保留 `category` 兼容字段；新的产品体验不应继续把 category 作为主要分组方式。
- 检查发布 Tab 中的程序检查器、AI 语义分析和一键修复 patch 预览是设计目标，当前后端基础 validation 只覆盖 schema、敏感信息和 Claude Code 编译。


## 11. 后续实现建议

优先级：

1. 服务端模型移除 category，迁移为 tags-only。
2. 将 Web 列表页的本地暂存上传升级为真实 ZIP / 文件夹上传 API，并与 Skill proposal 创建衔接。
3. 实现本地未发布草稿持久化。
4. 实现程序检查器。
5. 接入 AI 分析器。
6. 实现一键修复 patch 预览。
7. 实现 Agent 独立版本发布。
8. 继续完善 `skill-cli upload` 的多 Agent 校验与 Web 下载体验；`skill-cli install` registry 下载已经具备基础能力。
9. AI 配置页接入真实供应商、模型、连通性测试和用量统计。

当前前端 demo 可先保持本地 state / localStorage 模拟，作为交互验证版本。
